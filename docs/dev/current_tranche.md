# Current tranche

## Tranche 6 - Decouple the free-running synthetic tick from strand flush

Status: active; approved by maintainer 2026-07-19 ("Proceed with Finding 2 —
write the tranche entry first"), following the capture-path performance
investigation. Preceding tranche records live in
`docs/dev/completed_tranches/`.

### Purpose

Remove the Godot-main-thread stall imposed by `SyntheticProvider::advance()`'s
unconditional `strand_.flush()` on the free-running per-frame tick, without
weakening the deterministic host-stepped `advance_timeline()` contract that
verification scenes and publish-only checks depend on.

### Source-backed problem statement

* `CamBANGServer::_on_godot_tick()` (`src/godot/cambang_server.cpp`, the
  free-running per-frame path) calls
  `ProviderBroker::try_tick_virtual_time(dt_ns)`, which calls
  `SyntheticProvider::advance(dt_ns)`.
* `advance()` ends with an unconditional, blocking `strand_.flush()` — a
  cross-thread wait for the strand worker to deliver every queued
  provider->core callback — executed on the Godot main thread every frame.
* The flush exists for harness determinism: host-stepped
  `advance_timeline()` must deliver all callbacks generated by the advance
  before returning, so publish-only checks do not race queued frame delivery
  (comment at the flush site, mirroring shutdown()'s ordering).
* The free-running tick does not need that guarantee: Godot observes state
  only through tick-bounded snapshot publication, which occurs when facts
  integrate on the core thread regardless of when the strand delivers them.
  Scene 870 telemetry quantifies the cost environment this stall lives in
  (~19.8 s of synthetic frame rendering inside a ~79 s run, all on the Godot
  main thread).
* Capture relevance: rig submissions execute on the core thread and take
  `provider_state_mutex_`; shrinking the main-thread advance window reduces
  contention against `trigger_capture_submission()`.

### Scope

1. Add a `flush_strand` parameter (default `true`) to
   `SyntheticProvider::advance(uint64_t dt_ns, bool allow_paused_timeline_step,
   bool flush_strand)`. When `false`, skip only the terminal
   `strand_.flush()` (and its timing sample); all state mutation, timeline
   pumping, and frame emission behavior is unchanged.
2. Thread the same parameter through
   `ProviderBroker::try_tick_virtual_time(uint64_t dt_ns, bool flush_strand)`
   (default `true`).
3. `CamBANGServer::_on_godot_tick()`'s free-running per-frame call passes
   `flush_strand=false`. Every other caller — the host-stepped
   `advance_timeline()` path, smoke tools, verify-case harness — keeps the
   defaulted flushing behavior.

### Non-goals / scope guardrails

* Do not move synthetic frame rendering off the calling thread; that is a
  separate, larger refactor.
* Do not change `advance_timeline()` semantics, the strand's delivery order,
  event-class guarantees, or shutdown()'s flush/stop ordering.
* Do not change the locked Godot public API, snapshot schema, or publication
  counters.
* No behavior change for StubProvider.
* Core and SyntheticProvider remain platform-agnostic; the new parameter is
  pure control flow.

### Expected implementation files

* `src/imaging/synthetic/provider.h` / `provider.cpp`;
* `src/imaging/broker/provider_broker.h` / `provider_broker.cpp`;
* `src/godot/cambang_server.cpp` (one call site);
* this tranche record.

Any expansion beyond these files must be source-justified before editing.

### Accepted observable boundary

`CamBANGServer::_on_godot_tick()` refreshes CPU live display views immediately
after the tick's advance. With the flush skipped, a frame emitted by this
tick's advance may be integrated/retained after that refresh and therefore
become display-visible on the following tick instead of the same one. This is
within the tick-bounded observation model (same-tick delivery was never a
published guarantee, and under load frames already arrive asynchronously);
it is recorded here as a deliberate, bounded change, not an oversight. The
deferred-dispatch window in `try_tick_virtual_time()` is unaffected: timeline
request dispatches are produced synchronously inside `timeline_pump_()` on
the tick caller's thread, before the flush point, so no dispatch can migrate
to the strand thread.

### Acceptance criteria

* The free-running Godot tick no longer blocks on strand delivery; the
  host-stepped `advance_timeline()` path still flushes before returning.
* Default-parameter call sites compile unchanged and behave identically
  (defaults preserve flushing), so no smoke tool or verify-case harness edit
  is required.
* Deterministic publish-only verification scenes still pass: frames queued by
  a free-running tick are integrated by the core pump and published
  tick-bounded, with no missed-verdict, coalescing, or immutability
  regression.
* No Godot public binding change.
